1.3.4 Orientation

Content must not restrict its view and operation to a single display orientation — portrait or landscape — unless that specific orientation is essential. Users who have their devices mounted in a fixed orientation must be able to use the content.

Intent

Many users with motor disabilities mount their devices in a fixed position — a wheelchair-mounted tablet is often in landscape, while a communication device may be locked in portrait. If a web application forces a specific orientation, these users may be completely unable to use it.

This criterion requires that content work in both portrait and landscape orientations. Orientation restrictions should only be applied when it is genuinely essential to the content's function — a piano keyboard application that meaningfully requires landscape to show enough keys, for example.

The restriction may come from HTML (meta viewport with lock), CSS (@media (orientation:) hiding content), or JavaScript (detecting and refusing rotation). All of these are failures unless an essential exception applies.

Key Distinctions

OS Orientation Lock vs. App Lock
If a user has locked their device orientation in OS settings, the failure is the OS choice — not the website. This criterion covers when the content itself restricts orientation regardless of OS settings.
The Essential Exception
Orientation locking is permitted when a specific orientation is essential to the content. A bank cheque deposit scanner that must see the full cheque in landscape, a virtual piano app, or a specific game mechanic may qualify. These are narrow exceptions — most apps do not qualify.
Optimised vs. Locked
Content may be optimised for one orientation — showing a different layout in portrait vs. landscape — as long as it remains fully functional in both. The failure occurs only when content is completely blocked or broken in one orientation.

Examples

Example 1 of 4

CSS Orientation Lock — Content Hidden in Portrait

A web application hides its main content in portrait orientation and displays a "Please rotate your device" message. A user with a wheelchair-mounted tablet in landscape may be forced to use portrait. They see only the rotate message and cannot access the application.

Failure: F97
✗ Poor Practice
Portrait view — content blocked
Please rotate your device to landscape mode

✗ Users with fixed-mount devices cannot rotate — content inaccessible

CSSFAIL
/* F97: portrait orientation blocked */

@media (orientation: portrait) {
  .app-content {
    display: none;
  }
  .rotate-message {
    display: flex;
  }
}

/* Main content is completely
   hidden in portrait — not just
   optimised, but blocked -->
🔊A screen reader user in portrait hears"Please rotate your device to landscape mode." — entire application is inaccessible; no controls or content available

Problem (F97): Hiding all content in one orientation completely blocks access. Users with motor disabilities may have their devices mounted in a fixed orientation and cannot rotate them on demand.

✓ Good Practice — Both Orientations Work
Portrait view — content visible (different layout)
Dashboard

✓ Portrait layout — stacked, full function

CSSPASS
/* Content works in both orientations.
   Layout adapts — not blocks. */

/* Default: stacked portrait layout */
.dashboard {
  display: flex;
  flex-direction: column;
}

/* Landscape: side-by-side layout */
@media (orientation: landscape) {
  .dashboard {
    flex-direction: row;
  }
}

/* .rotate-message not used at all -->
🔊Screen reader user in portraitEntire dashboard is accessible; layout stacks vertically; all controls and content reachable via keyboard and screen reader

Fix: Remove the rotate-message pattern entirely. Use responsive CSS to adapt the layout for each orientation without hiding or blocking content. Both layouts must be fully functional.

Example 2 of 4

JavaScript Orientation Lock Using the Screen Orientation API

A web app calls screen.orientation.lock("landscape") programmatically, forcing the device into landscape regardless of how the user holds it. This prevents users with fixed mounts from ever accessing the content in portrait.

Failure: F97
✗ Poor Practice
JavaScriptFAIL
// F97: programmatic orientation lock

document.addEventListener("DOMContentLoaded", () => {
  // Forces landscape — blocks all
  // portrait-mounted device users
  screen.orientation
    .lock("landscape")
    .catch(err => {
      console.log("Could not lock", err);
    });
});

Problem (F97): The Screen Orientation API actively forces the device orientation. This removes user control. Even though the API may fail silently on some browsers, this pattern signals intent to lock orientation and should be avoided.

✓ Good Practice — No Locking
JavaScriptPASS
// Respond to orientation changes
// without locking them

window.addEventListener(
  "orientationchange",
  () => {
    // Adapt layout — don't lock
    updateLayout(
      screen.orientation.type
    );
  }
);

function updateLayout(orientationType) {
  // Adjust UI for orientation
  // but keep all content accessible
  const isLandscape =
    orientationType.includes("landscape");
  document.body.classList
    .toggle("landscape", isLandscape);
}

Fix: Listen to orientation changes and adapt the layout — but never call screen.orientation.lock() for general content. Let the user (or OS setting) control orientation. Only update CSS classes or layout logic in response to the change.

Example 3 of 4

Essential Exception — When Landscape Is Genuinely Required

A piano learning app renders a full piano keyboard requiring landscape to display the minimum number of keys needed to play the target exercises. The app legitimately requires landscape for its core function. However, it must communicate this clearly rather than silently blocking portrait users.

Essential exception
✗ Poor — Blocks Without Explanation
ScenarioFAIL
<!-- Even with a valid essential
     exception, simply hiding content
     without explanation fails.

     Bad pattern:
     - Portrait: blank screen
     - No message
     - No reason given
     - No alternative offered

     Users don't know if the app
     has crashed or requires rotation -->

Problem: Even genuine essential exceptions must communicate the requirement to users and explain why. A blank screen or silent failure is never acceptable.

✓ Good — Essential Exception with Clear Communication
HTMLACCEPTABLE
<!-- Essential exception handled well:
     1. Explains WHY landscape is needed
     2. Offers alternative if possible
     3. Screen reader accessible message -->

@media (orientation: portrait) {
  .piano-content { display: none; }
  .orientation-notice { display: block; }
}

<!-- In HTML: -->
<div class="orientation-notice"
     role="alert">
  <h2>Landscape required</h2>
  <p>This piano exercise needs landscape
  orientation to show all the keys.
  Please rotate your device or enable
  rotation in your OS settings.</p>
  <a href="exercises-text.html">
    View exercise as text instead
  </a>
</div>

Essential exception: If landscape is genuinely essential, explain why clearly in the portrait view, using a proper heading and descriptive text. Offer an alternative (text-based version). Ensure the notice itself is accessible via screen reader.

Example 4 of 4

Testing Orientation — Common Implementation Mistakes

Developers often encounter orientation failures unintentionally — forms that overflow in portrait, tables that clip, or navigation that becomes unusable. Testing both orientations systematically catches these issues before they reach users.

Testing guidance
✗ Common Unintentional Failures
CSS / HTMLCHECK THESE
/* Common 1.3.4 failures to check: */

/* 1. Fixed-width containers that
      overflow in portrait */
.app-wrapper {
  width: 1024px; /* too wide for portrait */
}

/* 2. Horizontal scrolling disabled */
body {
  overflow-x: hidden;
  overflow-y: scroll;
}

/* 3. Form fields with fixed widths */
input { width: 600px; }

/* 4. Viewport meta that disables zoom
      (related but separate issue) */
<meta name="viewport"
  content="width=device-width,
    initial-scale=1,
    user-scalable=no">
<!-- user-scalable=no also fails
     SC 1.4.4 Resize Text -->
✓ Responsive Patterns That Pass
CSSPASS
/* Responsive patterns that work in
   both orientations: */

/* 1. Fluid containers */
.app-wrapper {
  max-width: 1200px;
  width: 100%;
  box-sizing: border-box;
}

/* 2. Fluid inputs */
input, textarea {
  width: 100%;
  max-width: 480px;
}

/* 3. Viewport meta — allow zoom */
<meta name="viewport"
  content="width=device-width,
    initial-scale=1">
<!-- No user-scalable restriction -->

/* 4. Test using DevTools:
   Chrome/Firefox: toggle device
   toolbar → rotate icon -->

Testing approach: Use browser DevTools device emulation and toggle between portrait and landscape for every page. Check that all content is reachable, no horizontal overflow is clipped, and all interactive controls remain operable. Test with a screen reader in both orientations.
