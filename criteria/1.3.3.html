1.3.3 Sensory Characteristics

Instructions for using content must not rely solely on sensory characteristics — shape, colour, size, visual location, orientation, or sound. Instructions must include text identification that does not depend on the user being able to see, hear, or perceive spatial layout.

Intent

Instructions that say "click the green button" or "see the chart on the right" are meaningless to users who cannot see colour, cannot perceive layout, or are using a screen reader. This criterion requires that instructions identify UI components and content by name or label — not just by sensory properties.

The rule: instructions must work without seeing, hearing, or knowing where things are positioned.

Sensory characteristics include:

  • Shape: "click the circular icon", "press the square button"
  • Colour: "click the green button", "errors are shown in red"
  • Size: "click the large button", "see the smaller text below"
  • Visual location: "see the menu on the left", "the field on the right"
  • Orientation: "the arrow pointing down", "swipe left"
  • Sound: "wait for the beep", "when you hear the chime"

Examples

Example 1 of 4

Instruction References Colour Only

A form's help text tells users to "click the green Submit button." A blind user, or a user with colour blindness who cannot distinguish green, cannot identify which button is meant. The button must be identified by its label.

Failure: F14
✗ Poor Practice
Form with colour-only instruction

When you are ready, click the green button to submit your application.

✗ Colour is the only distinguishing instruction — fails for blind and colour-blind users

HTMLFAIL
<!-- F14: instruction relies solely
     on colour to identify the control -->
<p>When ready,
  click the green button
  to submit your application.
</p>
<button>Cancel</button>
<button class="btn-green">
  Submit
</button>
🔊A blind user hears"click the green button to submit" — no way to identify a "green button"; Cancel button and Submit button are both announced without colour

Problem (F14): Colour cannot be perceived by blind users or users with certain colour vision deficiencies. "Green button" is meaningless without a label name to identify it.

✓ Good Practice
Form with name-based instruction

When you are ready, click "Submit application" to complete your form.

✓ Button identified by its label — works for all users

HTMLPASS
<!-- Instruction uses the button label -->
<p>When ready, click
  "Submit application"
  to complete your form.
</p>
<button>Cancel</button>
<button class="btn-green">
  Submit application
</button>
🔊A blind user hears"click Submit application to complete your form" — matches the exact button label announced by the screen reader; unambiguous

Fix: Give the button a descriptive label ("Submit application") and reference that exact label in the instructions. The instruction works regardless of colour perception, vision, or screen reader usage.

Example 2 of 4

Instruction References Visual Location Only

An onboarding guide says "see the navigation menu on the left for all options." A screen reader user may encounter the nav anywhere in the DOM; they cannot find content by "left side." On a mobile device, the menu may not be on the left at all.

Failure: F14
✗ Poor Practice
HTMLFAIL
<!-- F14: location-only instruction -->
<p>
  Use the
  navigation menu on the left
  to explore all available sections.
</p>

<p>
  The error details can be found in
  the panel on the right side
  of your screen.
</p>

<!-- "On the left" / "on the right"
     are meaningless to:
     - Screen reader users
     - Mobile users (no columns)
     - Zoom users (layout changes) -->
🔊A screen reader user hears"Use the navigation menu on the left to explore all available sections." — cannot determine which element "on the left" refers to

Problem (F14): Spatial references like "on the left", "above", "below", and "on the right" assume sighted, 2D layout perception. They fail for screen reader users, keyboard-only users, and anyone using a single-column or responsive layout.

✓ Good Practice
HTMLPASS
<!-- Reference by name/label instead -->
<p>
  Use the
  Main navigation menu
  to explore all available sections.
</p>

<p>
  The error details can be found in
  the Error details panel.
</p>

<!-- Combine with location for
     sighted users — this is fine: -->
<p>
  Use the Main navigation menu
  (on the left in desktop view)
  to explore all sections.
</p>
🔊A screen reader user hears"Use the Main navigation menu to explore all available sections." — matches the aria-label of the nav element; unambiguous without spatial context

Fix: Reference UI elements by their label or name. The label used in the instruction should match what the element is called — its heading, aria-label, or visible text. Location may be included as supplementary context but must not be the only identifier.

Example 3 of 4

Icon Buttons Referenced Only by Shape

A media player's help documentation says "click the triangular play button." A blind user cannot perceive a triangle. If the button has no accessible label — just an icon — the instruction fails doubly: the sensory reference is meaningless and the button itself is inaccessible.

Failure: F14 SC 1.1.1 related
✗ Poor Practice
Media player controls — unlabelled icons

Click the triangular button to play. Click the two vertical bars to pause.

HTMLFAIL
<!-- F14: shape-only reference
     + SC 1.1.1 failure (no label) -->
<p>Click the
  triangular button to play.
</p>

<!-- Button has no accessible name -->
<button>
  <svg>…play icon…</svg>
</button>
🔊Screen reader announces"button" (no label) — the instruction "triangular button" is meaningless; the button itself has no accessible name
✓ Good Practice
Same controls — labelled and referenced by name

Click "Play" to start the audio. Click "Pause" to stop.

HTMLPASS
<!-- Instruction uses button label.
     Button has accessible name via
     aria-label (also fixes SC 1.1.1). -->
<p>Click
  "Play" to start the audio.
</p>

<button aria-label="Play">
  <svg aria-hidden="true">
    …play icon…
  </svg>
</button>
🔊Screen reader announces"Play, button" — matches the instruction "Click Play to start the audio"; accessible and unambiguous

Fix: Add aria-label to icon-only buttons. Update instructions to reference the label name. This also fixes SC 1.1.1 (non-text content) and SC 4.1.2 (name, role, value) simultaneously.

Example 4 of 4

Combining Sensory Cues with Text Labels — Acceptable

Instructions may include sensory characteristics in addition to a text label — they just cannot be the only means of identification. "Click the green Submit button" fails. "Click the Submit button (green, bottom right)" passes — the label "Submit" is sufficient on its own, and the sensory details are supplementary for sighted users.

Best practice
✗ Sensory Only — Fails
Examples that failFAIL
<!-- All fail 1.3.3 -->

"Click the green button."
<!-- colour only -->

"Press the large button."
<!-- size only -->

"Select the round option."
<!-- shape only -->

"Use the field on the right."
<!-- location only -->

"Click when you hear the chime."
<!-- sound only -->

"See the diagram below."
<!-- location only -->
✓ Label + Sensory Cue — Passes
Examples that passPASS
<!-- All pass 1.3.3 -->

"Click the Submit button."
<!-- label only — sufficient -->

"Click the Submit button (green)."
<!-- label + colour as supplement -->

"Press the Confirm button (large,
 at the bottom of the form)."
<!-- label + size + location -->

"Use the Postcode field."
<!-- label — no location needed -->

"See the 'Regional breakdown' chart."
<!-- named reference — no "below" -->

"When the Ready indicator turns green,
 click Start."
<!-- named indicator + colour as
     supplement. "Start" is the label -->

Rule: The instruction must be understandable without the sensory cue. Test by removing all colour/shape/location/sound references — if the instruction still makes sense, it passes. If it becomes ambiguous, add a text label.
