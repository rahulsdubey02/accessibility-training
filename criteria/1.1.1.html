Level A WCAG 2.0 1.1 Text Alternatives

1.1.1 Non-text Content

All non-text content that is presented to the user has a text alternative that serves the equivalent purpose — with specific exceptions for controls, time-based media, tests, sensory experiences, CAPTCHA, and decoration.

Intent

The purpose of this criterion is to make information conveyed by non-text content accessible through the use of a text alternative. Text alternatives can be rendered through any sensory modality — visual, auditory, or tactile — to match the needs of the user.

When a text alternative is not provided, screen reader users receive no information, or worse, a filename or placeholder that conveys nothing meaningful. This is one of the most commonly violated — and most impactful — accessibility requirements.

Exceptions & Special Cases

Controls / Inputs
Non-text content acting as a control needs a name that describes its purpose (not the image itself).
Time-based Media
Provides descriptive identification only. Full alternatives are covered by 1.2.x criteria.
Tests / Exercises
Where describing the content would invalidate the assessment, provide only a descriptive name.
Sensory Experiences
Content primarily creating a sensory experience (e.g., music) needs a descriptive text alternative.
CAPTCHA
Must identify it as a CAPTCHA and provide alternatives using different sensory modalities.
Decoration / Invisible
Purely decorative images must be implemented so assistive technology can ignore them entirely.

Examples

Example 1 of 7

Informative Image

A company logo displayed in the site header — it identifies whose website the user is on. Without an alt attribute, screen readers announce the filename or nothing at all.

Failure: F65 Technique: H37
✗ Poor Practice
Rendered output
HTML FAIL 
<img src="acme-logo.png">

<!-- alt attribute is completely missing -->
🔊 Screen reader announces "acme-logo.png, graphic"

Problem: The screen reader reads the filename as the label. Users have no idea what organisation this logo represents.

✓ Good Practice
Rendered output
HTML PASS 
<img
  src="acme-logo.png"
  alt="Acme Corp"
>
🔊 Screen reader announces "Acme Corp, graphic"

Fix: Add alt text that conveys the same information as the image — for a logo, the company name is sufficient.

Remediation Notes

💡

Alt text should convey purpose, not appearance

Write alt="Acme Corp" not alt="blue rounded rectangle with white text". The goal is equivalent information, not description.

🔍

Test with automated tools first

Missing alt attributes are easy to detect with axe, WAVE, or Lighthouse. Catch these programmatically before manual testing.

✏️

Keep it concise

Screen readers pause at punctuation. Most alt text should be one short phrase. Avoid starting with "Image of…" — the element type is already announced.

Example 2 of 7

Decorative Image

A visual flourish or section divider that adds aesthetic value but carries no informational content. The wrong treatment forces screen reader users to hear meaningless noise on every page.

Failure: F38 Failure: F39 Technique: H67
✗ Poor Practice
Rendered output A decorative wave separator
HTML FAIL 
<!-- Case A: missing alt (F38) -->
<img src="wave.svg">

<!-- Case B: unnecessary alt text (F39) -->
<img src="wave.svg"
     alt="decorative wavy divider"
>
🔊 Screen reader announces (Case A) "wave.svg, graphic"

Problem A: No alt causes the filename to be read aloud — meaningless noise.

Problem B: A non-empty alt on a decorative image forces screen readers to announce it, interrupting the reading experience unnecessarily.

✓ Good Practice
Rendered output (visually identical) Screen readers skip this entirely
HTML PASS 
<!-- Null alt — AT ignores the image -->
<img
  src="wave.svg"
  alt=""
  role="presentation"
>

<!-- Better: CSS background (C9) -->
<div class="wave-divider"
     aria-hidden="true"></div>
🔊 Screen reader announces (nothing — image is skipped)

Fix: Use alt="" (empty string, not missing) so assistive technology knows the image is intentionally skipped. Pair with role="presentation" for maximum compatibility, or use CSS backgrounds for decoration instead.

Remediation Notes

⚠️

Empty alt ≠ missing alt

alt="" is a deliberate signal. A missing alt attribute triggers different (often broken) behaviour in assistive technology.

🎨

Prefer CSS for decoration

Moving purely decorative images to CSS background-image removes them from the accessibility tree entirely (Technique C9).

Example 3 of 7

Functional Image — Icon Button

A search button that contains only an icon and no visible text. When the image has no accessible name, a screen reader user knows there is a button but not what it does.

Failure: F65 Technique: H36 Technique: ARIA6
✗ Poor Practice
Rendered output No visible label — and no accessible name either
HTML FAIL 
<button>
  <img src="search-icon.svg">
</button>

<!-- img has no alt — button has no name -->
🔊 Screen reader announces "button"

Problem: The button has no accessible name. A screen reader user hears "button" with no indication of what it does.

✓ Good Practice
Option A — alt on the image Visually same — now has accessible name "Search"
HTML PASS 
<!-- Option A: alt on the image -->
<button>
  <img src="search-icon.svg"
       alt="Search">
</button>

<!-- Option B: aria-label on button (SVG icon) -->
<button aria-label="Search">
  <svg aria-hidden="true" …>…</svg>
</button>
🔊 Screen reader announces "Search, button"

Fix: For <img> inside a button, add alt="Search". For SVG icons, hide the SVG with aria-hidden="true" and label the button with aria-label.

Remediation Notes

🏷️

The name comes from the control, not the image

For interactive controls, the accessible name should describe what the control does, not what the icon looks like. "Submit" not "pointing arrow".

🔗

Links with image + text

If a link wraps both an image and text, use alt="" on the image — the link text already provides the name (Technique H2).

📐

Don't double-announce

If a button has aria-label="Search", the SVG inside should be aria-hidden="true". Otherwise screen readers may announce both.

Example 4 of 7

Complex Image — Chart or Infographic

A bar chart showing quarterly revenue. A vague alt like "Chart" is a named failure (F30) — it's present but useless. The text alternative must convey the same information as the image.

Failure: F30 Technique: G92 Technique: G73
✗ Poor Practice
Rendered output
HTML FAIL 
<!-- Case A: no alt (F65) -->
<img src="revenue.png">

<!-- Case B: vague alt (F30) -->
<img src="revenue.png"
     alt="Chart">
🔊 Screen reader announces (Case B) "Chart, graphic"

Problem: "Chart" tells the user nothing about the data. Failure F30 specifically names placeholder text that is not an actual alternative.

✓ Good Practice
Rendered output with figure + caption
HTML PASS 
<figure>
  <img
    src="revenue.png"
    alt="2024 Revenue: Q1 $1.8M,
         Q2 $2.1M, Q3 $1.5M,
         Q4 $2.4M"
  >
  <!-- Long data: link to a data table -->
  <figcaption>
    <a href="#revenue-table">
      View full data table
    </a>
  </figcaption>
</figure>
🔊 Screen reader announces "2024 Revenue: Q1 $1.8M, Q2 $2.1M, Q3 $1.5M, Q4 $2.4M, graphic"

Fix: For short data (4 values), encode it in alt. For complex charts, provide a summary in alt and link to a data table or long description (Technique G73).

Remediation Notes

📊

When alt alone isn't enough

For complex charts, use alt for a summary trend, then link to a fully accessible data table (<table>) nearby on the page.

🔲

SVG charts can be made natively accessible

Inline SVG charts can use <title>, <desc>, and ARIA roles to expose data to screen readers without needing a separate table.

❌

Watch out for Failure F30

"Chart", "Graph", "Image001.png" — these are explicitly named failures. An alt attribute that is present but not informative still fails 1.1.1.

Example 5 of 7

Inline SVG Icon

Navigation icons rendered as inline SVG. Unlike <img>, inline SVG is not automatically exposed as an image to screen readers — it requires explicit ARIA markup.

Failure: F65 Technique: ARIA6 Technique: ARIA10
✗ Poor Practice
Rendered output (navigation icons)
HTML FAIL 
<nav>
  <a href="/">
    <svg viewBox="0 0 24 24">
      <path d="M3 9l9-7…"/>
    </svg>
  </a>
</nav>

<!-- No title, no aria-label anywhere -->
🔊 Screen reader announces "link" (no name)

Problem: Inline SVG without labels produces unlabelled links. A keyboard user navigating links hears only "link, link, link" with no way to distinguish them.

✓ Good Practice
Visually identical — now accessible
HTML PASS 
<nav>
  <a href="/" aria-label="Home">
    <svg aria-hidden="true"
         viewBox="0 0 24 24">
      <path d="M3 9l9-7…"/>
    </svg>
  </a>
</nav>

<!-- OR: title element inside SVG -->
<svg role="img"
     aria-labelledby="home-title">
  <title id="home-title">Home</title>
  <path d="M3 9l9-7…"/>
</svg>
🔊 Screen reader announces "Home, link"

Fix: Add aria-label to the parent link and hide the SVG with aria-hidden="true". Alternatively, add a <title> inside the SVG and reference it with aria-labelledby.

Remediation Notes

🙈

Hide SVG from AT when the parent is labelled

If the containing link or button has an aria-label, set aria-hidden="true" on the SVG to prevent double-announcing.

📝

Visually hidden text is a valid alternative

A <span class="sr-only">Home</span> adjacent to the SVG is semantically clean and well-supported across all screen readers.

Example 6 of 7

Image of Text

A stylised call-to-action button or heading rendered as a raster image. Screen readers cannot extract the text from the pixels — the alt attribute must exactly match what the image says.

Failure: F65 Failure: F30 Technique: H37
✗ Poor Practice
Rendered output
HTML FAIL 
<!-- Case A: No alt (F65) -->
<img src="get-started.png">

<!-- Case B: wrong/vague alt (F30) -->
<img src="get-started.png"
     alt="button">
🔊 Screen reader announces (Case B) "button, graphic"

Problem: Neither the filename nor "button" conveys the marketing copy in the image. Screen reader users miss the call to action entirely.

✓ Good Practice
Option A — match the text in the image Better: replace with real CSS text
HTML PASS 
<!-- Match alt text exactly to image text -->
<img
  src="get-started.png"
  alt="Get started today"
>

<!-- Preferred: use real text + CSS styling -->
<a href="/signup"
   class="btn btn--gradient">
  Get started today
</a>
🔊 Screen reader announces "Get started today, graphic"

Fix: The alt attribute must match the text in the image verbatim. The best fix is to eliminate image-of-text entirely and use real HTML text styled with CSS — this also supports text resize (SC 1.4.4) and user stylesheets.

Remediation Notes

🎯

Alt must match image text exactly

If the image says "Get started today — free for 30 days", the alt must say exactly that. Paraphrasing is Failure F30.

✨

CSS can achieve almost any text styling

Custom fonts (Google Fonts, variable fonts), gradients on text, letter-spacing, and drop shadows mean there is almost never a reason to use images of text.

↕️

Images of text fail other criteria too

They can't be resized (1.4.4), are not overridable with user stylesheets, and may fail colour contrast if the image is a JPEG (lossy compression).

Example 7 of 7

CAPTCHA

A visual CAPTCHA used to verify the user is human. This is a special exception: alt text must not solve the CAPTCHA, but it must identify its purpose and a non-visual alternative must be provided (Techniques G143 + G144).

Exception case Technique: G143 Technique: G144
✗ Poor Practice
Rendered output
No audio alternative. No helpful alt text.
HTML FAIL 
<!-- Case A: no alt -->
<img src="captcha.png">

<!-- Case B: vague alt, no alternative -->
<img src="captcha.png"
     alt="security check">
<!-- No audio/alternative mode offered -->
🔊 Screen reader announces (Case B) "security check, graphic" — cannot complete form

Problem: Screen reader users cannot solve the CAPTCHA and cannot submit the form. They are blocked from the service entirely.

✓ Good Practice
Rendered output with audio alternative
HTML PASS 
<!-- G143: describe purpose, not the answer -->
<img
  src="captcha.png"
  alt="Visual CAPTCHA: enter the
       characters shown to verify
       you are not a robot"
>

<!-- G144: offer an alternative modality -->
<button type="button">
  Use audio CAPTCHA instead
</button>
🔊 Screen reader announces "Visual CAPTCHA: enter the characters shown to verify you are not a robot, graphic" — then finds audio alternative button

Fix (G143): Describe the purpose of the CAPTCHA — never reveal the answer.

Fix (G144): Provide an alternative CAPTCHA using a different sense (audio, logic puzzle). Both G143 and G144 are required together.

Remediation Notes

🔐

Both G143 and G144 are required

G143 alone (alt text describing the CAPTCHA) is not sufficient. A second CAPTCHA using a different sensory modality (G144) must also be available.

🧩

Consider cognitive accessibility alternatives

Third-party CAPTCHA providers (e.g., hCaptcha, Friendly CAPTCHA) offer logic-based and invisible alternatives that reduce friction for all users.

🚫

Never solve the CAPTCHA in alt text

The alt text must describe the CAPTCHA's purpose, not its answer. Exposing the answer defeats the security mechanism.
