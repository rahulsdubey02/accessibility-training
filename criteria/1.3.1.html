1.3.1 Info and Relationships

Information, structure, and relationships conveyed through visual presentation can be programmatically determined or are available in text. If something looks like a heading, acts like a table header, or appears required — the markup must express that meaning.

Intent

Sighted users perceive structure visually — headings look larger and bolder, table headers appear different from data cells, required fields are marked with an asterisk. This criterion requires that the same structural meaning be available programmatically — encoded in the HTML so that assistive technologies (screen readers, braille displays) can expose it to users who cannot see the visual presentation.

The key principle: if it looks like structure, it must be marked up as structure.

  • Text styled to look like a heading → must be a <h1>–<h6> element
  • A row of cells acting as column headers → must use <th> not <td>
  • Fields grouped together → must use <fieldset> + <legend>
  • Required status conveyed by red asterisk → must also be in the label or via ARIA

Examples

Example 1 of 5

Data Table with No Header Cells

A pricing table displays plan names, monthly costs, and feature limits. Visually the top row clearly acts as column headers. But the HTML uses only <td> throughout — screen readers announce data cells without any column association.

Failure: F91 Technique: H51
✗ Poor Practice
Rendered output
Plan Price/mo Users
Starter £9 1–5
Pro £29 Up to 20

✗ Header row looks right but has no semantic markup

HTMLFAIL
<table>
  <tr>
    <!-- F91: header row uses <td>,
         not <th> -->
    <td>Plan</td>
    <td>Price/mo</td>
    <td>Users</td>
  </tr>
  <tr>
    <td>Starter</td>
    <td>£9</td>
    <td>1–5</td>
  </tr>
</table>
🔊Screen reader announces"Table, 3 columns, 3 rows. Row 1: Plan, Price/mo, Users. Row 2: Starter, £9, 1–5." — no column-header relationship established

Problem (F91): When navigating individual cells, a screen reader cannot tell the user which column header a cell belongs to. "£9" has no relationship to "Price/mo" in the markup.

✓ Good Practice
Same visual result — correct markup
Plan Price/mo Users
Starter £9 1–5
Pro £29 Up to 20

✓ <th> elements with <thead> establish column relationships

HTMLPASS
<!-- H51: use <th> for header cells
     scope attribute clarifies direction -->
<table>
  <caption>Pricing plans</caption>
  <thead>
    <tr>
      <th scope="col">Plan</th>
      <th scope="col">Price/mo</th>
      <th scope="col">Users</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Starter</td>
      <td>£9</td>
      <td>1–5</td>
    </tr>
  </tbody>
</table>
🔊Screen reader announces"Table, Pricing plans, 3 columns, 2 rows. Row 1, Column 1: Starter. Price/mo: £9. Users: 1–5." — column headers announced with each cell

Fix (H51): Use <th scope="col"> for column headers and <th scope="row"> for row headers. Wrap header rows in <thead> and data rows in <tbody>. Add a <caption> for the table's purpose.

Example 2 of 5

Visual Heading Implemented with a <div>

A developer styles a <div> to look like a section heading — large, bold, and dark. Visually it functions as a heading. A screen reader user navigating by headings (a very common pattern) skips it entirely because it is not an <h1>–<h6> element.

Failure: F2 Technique: H42
✗ Poor Practice
Rendered output — looks like a heading
Account Settings

Manage your profile, security, and notification preferences.

✗ Looks like an h2 but is a <div> — invisible to heading navigation

HTMLFAIL
<!-- F2: heading conveyed visually
     but not in markup -->
<div class="heading-styled">
  Account Settings
</div>
<p>Manage your profile…</p>

/* CSS only creates visual appearance */
.heading-styled {
  font-size: 1.4rem;
  font-weight: 700;
  border-bottom: 2px solid blue;
}
🔊When navigating by headings (H key)"Account Settings" is skipped entirely — the section cannot be reached via heading navigation and has no heading level announced

Problem (F2): Screen reader users frequently navigate a page by jumping between headings using a keyboard shortcut. A <div> styled to look like a heading is completely invisible to this navigation pattern.

✓ Good Practice
HTMLPASS
<!-- H42: use actual heading element.
     CSS still controls appearance.
     Structure comes from the tag. -->
<h2 class="section-heading">
  Account Settings
</h2>
<p>Manage your profile…</p>

/* Style the h2 — same visual result */
.section-heading {
  font-size: 1.4rem;
  font-weight: 700;
  border-bottom: 2px solid blue;
}
🔊Screen reader announces"Account Settings, heading level 2" — reachable via heading navigation; the section is now navigable

Fix (H42): Replace the <div> with the appropriate heading level element (<h2>–<h6>). CSS still controls the visual appearance. The heading level should reflect the document outline, not the desired size.

Example 3 of 5

Form Fields Without Programmatically Associated Labels

A login form uses placeholder text as the only labelling mechanism. The placeholder disappears when the user starts typing, leaving the field unlabelled. Screen readers announce placeholder text inconsistently and it vanishes from view — both sighted and non-sighted users are affected.

Failure: F68 Technique: H44
✗ Poor Practice
Login form — placeholders as labels

✗ Placeholder disappears when user types — label is lost

HTMLFAIL
<!-- F68: no <label> associated -->
<input
  type="email"
  placeholder="Email address"
>
<input
  type="password"
  placeholder="Password"
>

<!-- placeholder is not a label:
     - Disappears on focus/input
     - Often low contrast
     - Not consistently exposed by AT -->
🔊Screen reader on focus"Edit text" (placeholder text may or may not be read — varies by AT). Once user starts typing: field has no label; "edit blank"
✓ Good Practice
Form with visible labels

✓ Visible label always present, programmatically associated

HTMLPASS
<!-- H44: <label for> associates
     label with input by id -->
<div>
  <label for="email">Email address</label>
  <input
    id="email"
    type="email"
    autocomplete="email"
  >
</div>
<div>
  <label for="password">Password</label>
  <input
    id="password"
    type="password"
    autocomplete="current-password"
  >
</div>
🔊Screen reader on focus"Email address, edit text" — label is permanent, does not disappear when the user types

Fix (H44): Use <label for="id"> paired with a matching id on the input. The label remains visible and associated permanently. Use placeholder text as supplementary hint only, never as the only label.

Example 4 of 5

Required Fields Indicated by Visual Asterisk Only

A registration form marks required fields with a red asterisk (*). The key explaining "* = required" appears at the top of the form. The asterisk is a visual character inside the label — screen readers announce "asterisk" or ignore it, and the programmatic required state is never communicated.

Failure: F81 Technique: H90
✗ Poor Practice
HTMLFAIL
<!-- F81: required communicated
     only through visual asterisk -->
<p>Fields marked * are required</p>

<label for="name">
  Full name <span aria-hidden="true">*</span>
</label>
<input id="name" type="text">

<!-- No required attribute, no
     aria-required, no text cue
     in the accessible name -->
🔊Screen reader on focus"Full name, edit text" — required status is not communicated; the asterisk is hidden; user may submit without completing required fields

Problem (F81): Using aria-hidden="true" on the asterisk hides it from screen readers — but then no mechanism communicates required status. The asterisk must be paired with required or aria-required.

✓ Good Practice
HTMLPASS
<!-- H90: visual asterisk + programmatic
     required attribute together -->
<p>Fields marked * are required</p>

<label for="name">
  Full name
  <span aria-hidden="true"> *</span>
</label>
<input
  id="name"
  type="text"
  required
  <!-- or: aria-required="true" -->
>
🔊Screen reader on focus"Full name, edit text, required" — required status announced regardless of the asterisk

Fix (H90): Add the required attribute (or aria-required="true") to every required field. The visual asterisk is kept for sighted users with aria-hidden="true" to avoid "asterisk" being read aloud. Both audiences receive the required cue through their own channel.

Example 5 of 5

Visual List Built with Line Breaks and Dashes

A features section presents a list of product capabilities using CSS-styled <div> elements or paragraphs with manual dashes — "— Real-time sync", "— 256-bit encryption". Screen readers cannot determine how many items are in the list or jump between them.

Failure: F43 Technique: H48
✗ Poor Practice
HTMLFAIL
<!-- F43: list structure conveyed
     only through visual formatting -->
<div class="features">
  <p>— Real-time sync</p>
  <p>— 256-bit encryption</p>
  <p>— 99.9% uptime SLA</p>
  <p>— 24/7 support</p>
</div>
🔊Screen reader reads"dash Real-time sync. dash 256-bit encryption. dash 99.9% uptime SLA…" — no list count announced, no list navigation; "dash" is read aloud

Problem (F43): Screen readers cannot tell the user there are 4 items in this group. The visual "—" marker is read literally. Users cannot jump between list items using arrow keys.

✓ Good Practice
HTMLPASS
<!-- H48: use <ul> for unordered lists.
     CSS can style the bullets.
     Structure communicates the group. -->
<ul>
  <li>Real-time sync</li>
  <li>256-bit encryption</li>
  <li>99.9% uptime SLA</li>
  <li>24/7 support</li>
</ul>
🔊Screen reader announces"List, 4 items. Real-time sync. 256-bit encryption. 99.9% uptime SLA. 24/7 support. End of list." — count and list structure communicated

Fix (H48): Use <ul> for unordered items and <ol> for sequential items. Use CSS list-style to control bullet appearance. The HTML element type communicates the list structure semantically.
