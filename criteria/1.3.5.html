1.3.5 Identify Input Purpose

The purpose of each input field that collects personal information about the user can be programmatically determined. This enables browsers to offer accurate autofill, assistive technologies to display custom icons, and users to understand what each field is collecting.

Intent

The autocomplete attribute tells browsers — and assistive technologies — what type of personal information a field collects. When correctly applied, browsers can pre-populate fields with saved data, reducing cognitive load and the need for typing. This particularly benefits users with cognitive disabilities, motor disabilities, and those using switch access or eye-tracking.

This criterion applies specifically to fields collecting personal data about the current user — name, email, phone, address, payment info, and so on. It uses the WHATWG autocomplete token list defined in the HTML specification.

Important constraints:

  • Only applies to inputs collecting personal information (not search, filter, or other inputs)
  • The input must be in a <form> (directly or via form attribute)
  • The autocomplete token must be a value from the official WHATWG list — custom values do not satisfy the criterion
  • Does not apply when the input's purpose is intentionally ambiguous (security questions)

Key autocomplete Tokens

Personal Identity
name · given-name · family-name · additional-name · nickname · honorific-prefix · honorific-suffix · organization · organization-title
Contact Details
email · tel · tel-national · url · username
Address
street-address · address-line1 · address-line2 · address-level1 (state) · address-level2 (city) · postal-code · country · country-name
Payment & Credentials
cc-name · cc-number · cc-exp · cc-exp-month · cc-exp-year · cc-csc · current-password · new-password · one-time-code

Examples

Example 1 of 4

Personal Data Fields Without autocomplete

A registration form collects name, email, and phone number. None of the fields have an autocomplete attribute. Browsers cannot offer autofill. A user with motor difficulties who saves data in their browser's autofill must type everything manually every time.

Failure: F107 Technique: H98
✗ Poor Practice
Registration form — no autofill available

✗ Browser cannot identify field purposes — no autofill dropdown offered

HTMLFAIL
<!-- F107: no autocomplete attribute
     on personal data inputs -->
<form>
  <label for="name">Full name</label>
  <input id="name" type="text">

  <label for="email">Email address</label>
  <input id="email" type="email">

  <label for="phone">Phone number</label>
  <input id="phone" type="tel">
</form>

Problem (F107): Without autocomplete, browsers cannot identify what each field expects. Users with motor or cognitive disabilities who rely on autofill must type all personal data manually on every form.

✓ Good Practice
Same form — autofill available
✓

✓ Browser autofill populates all fields — one click for all data

HTMLPASS
<!-- H98: autocomplete tokens applied
     to each personal data field -->
<form>
  <label for="name">Full name</label>
  <input id="name" type="text"
         autocomplete="name">

  <label for="email">Email address</label>
  <input id="email" type="email"
         autocomplete="email">

  <label for="phone">Phone number</label>
  <input id="phone" type="tel"
         autocomplete="tel">
</form>

Fix (H98): Add the correct autocomplete token to each personal data field. The browser recognises the field's purpose and can offer autofill from saved profiles — one click populates the entire form.

Example 2 of 4

Full Address Form with Complete autocomplete Tokens

A checkout form collects a full delivery address. Without autocomplete, every field must be manually typed. With correct tokens, the browser fills the entire shipping address from the user's saved profile in a single interaction.

Technique: H98
✗ Poor Practice
HTMLFAIL
<form>
  <input type="text"
         placeholder="Address line 1">
  <input type="text"
         placeholder="Address line 2">
  <input type="text"
         placeholder="City">
  <input type="text"
         placeholder="Postcode">
  <input type="text"
         placeholder="Country">
</form>

<!-- No autocomplete attributes,
     no labels, no visible context.
     Also fails SC 1.3.1 (no labels) -->

Problem: No autocomplete attributes — browser cannot autofill. No <label> elements — also fails SC 1.3.1. A user entering their 10th checkout form this month must type 5+ fields every time.

✓ Full Address Form — Correct Tokens
HTMLPASS
<form>
  <div>
    <label for="addr1">Address line 1</label>
    <input id="addr1" type="text"
           autocomplete="address-line1">
  </div>
  <div>
    <label for="addr2">Address line 2</label>
    <input id="addr2" type="text"
           autocomplete="address-line2">
  </div>
  <div>
    <label for="city">City</label>
    <input id="city" type="text"
           autocomplete="address-level2">
  </div>
  <div>
    <label for="postcode">Postcode</label>
    <input id="postcode" type="text"
           autocomplete="postal-code">
  </div>
  <div>
    <label for="country">Country</label>
    <select id="country"
            autocomplete="country">
      <option value="GB">United Kingdom</option>
    </select>
  </div>
</form>

Fix (H98): Apply the correct WHATWG autocomplete token to each field. Note: city maps to address-level2 (not city), and country should use the ISO 3166 country code token country. autocomplete works on <select> elements too.

Example 3 of 4

Wrong or Custom autocomplete Token

A developer adds autocomplete attributes but uses non-standard values ("firstname", "lastname") instead of the WHATWG tokens ("given-name", "family-name"). Non-standard values are treated as off by browsers — no autofill is offered.

Failure: F107 Technique: H98
✗ Invalid Tokens
HTMLFAIL
<!-- F107: custom tokens that do NOT
     match the WHATWG autocomplete
     token list -->
<input type="text"
       autocomplete="firstname">
<!-- should be: given-name -->

<input type="text"
       autocomplete="lastname">
<!-- should be: family-name -->

<input type="text"
       autocomplete="postcode">
<!-- should be: postal-code -->

<input type="text"
       autocomplete="phone">
<!-- should be: tel -->

Problem (F107): Non-standard autocomplete values ("firstname", "postcode") are not recognised by browsers and do not satisfy 1.3.5. The field purpose cannot be programmatically determined. Always use the exact WHATWG token list values.

✓ Correct WHATWG Tokens
HTMLPASS
<!-- H98: exact WHATWG tokens -->
<input type="text"
       autocomplete="given-name">
<!-- NOT: firstname -->

<input type="text"
       autocomplete="family-name">
<!-- NOT: lastname -->

<input type="text"
       autocomplete="postal-code">
<!-- NOT: postcode -->

<input type="tel"
       autocomplete="tel">
<!-- NOT: phone -->

Fix: Use the exact token values from the WHATWG HTML specification. Common mistakes: firstname → given-name, postcode → postal-code, phone → tel, card-number → cc-number.

Example 4 of 4

Separate Billing and Shipping Sections Using section- Prefix

A checkout form has both a shipping address and a billing address. Both sections contain identical-looking fields (address line 1, city, postcode). The section- prefix distinguishes them so the browser offers the correct autofill for each group.

Technique: H98 Advanced pattern
✗ Ambiguous — Same Token, Two Purposes
HTMLAMBIGUOUS
<!-- Browser cannot distinguish
     which address-line1 is shipping
     vs. billing. Autofill may apply
     the same data to both fields. -->

<fieldset>
  <legend>Shipping address</legend>
  <input autocomplete="address-line1">
  <input autocomplete="postal-code">
</fieldset>

<fieldset>
  <legend>Billing address</legend>
  <input autocomplete="address-line1">
  <input autocomplete="postal-code">
</fieldset>

Issue: Both sets of fields have the same autocomplete token. The browser may autofill both with the same address, or behave unexpectedly. The section is not clearly distinguished.

✓ Section Prefix Distinguishes Groups
HTMLPASS
<!-- The "section-" prefix groups
     related fields. The browser
     treats each section as a
     separate autofill context. -->

<fieldset>
  <legend>Shipping address</legend>
  <input
    autocomplete="section-shipping
      address-line1">
  <input
    autocomplete="section-shipping
      postal-code">
</fieldset>

<fieldset>
  <legend>Billing address</legend>
  <input
    autocomplete="section-billing
      address-line1">
  <input
    autocomplete="section-billing
      postal-code">
</fieldset>
🔊Browser autofill behaviourShipping fields offer shipping address from saved profile. Billing fields offer billing address separately — the section prefix allows different saved addresses per group

Pattern: The section-* prefix (any unique string after "section-") groups autocomplete fields. Within each section, use standard WHATWG tokens. The section name is arbitrary — use meaningful names like "section-shipping" or "section-billing".
