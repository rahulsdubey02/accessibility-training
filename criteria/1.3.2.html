1.3.2 Meaningful Sequence

When the sequence in which content is presented affects its meaning, the correct reading order can be programmatically determined. The DOM order — not the visual layout — is what assistive technologies and keyboard users experience.

W3C Understanding 1.3.2 ↗
Criterion
1.3.2
Level
A (Minimum)
Principle
1. Perceivable
WCAG Version
2.0 (2008)
Examples below
4 scenarios

Intent

Sighted users perceive layout visually — a multi-column layout reads left column first, then right. CSS can reorder content visually without changing the DOM order. Screen readers and keyboard navigation follow the DOM order, not the visual layout.

This criterion requires that when reading order affects meaning — a numbered list, a question followed by an answer, a form with instructions before fields — the DOM order must match the correct logical reading order. Using CSS to change visual position without changing DOM order creates a mismatch.

This interacts closely with SC 2.4.3 (Focus Order) — both require that the logical sequence be correct. 1.3.2 focuses on reading/perception; 2.4.3 focuses on focus/interaction.

Examples

Example 1 of 4

CSS position: absolute Reverses Reading Order

A two-column layout places a sidebar on the left visually, but the DOM has the main content first and the sidebar second. CSS uses absolute positioning to move the sidebar into the left column. Screen readers follow the DOM and announce main content before sidebar — matching visual intent. But when the same technique reverses intended reading order, it fails.

Failure: F32 Technique: C27
✗ Poor Practice
Visual layout (left to right)

Step 1

Read the instructions first

Step 2

Complete the form

Visual: Step 1 → Step 2 (correct)

✗ DOM order: Step 2 → Step 1 (wrong)

HTML + CSSFAIL
<!-- DOM order: Step 2 first,
     then Step 1 — wrong! -->
<div style="position:relative">
  <div style="position:absolute;
    left:200px">
    Step 2: Complete the form
  </div>
  <div style="position:absolute;
    left:0">
    Step 1: Read the instructions
  </div>
</div>

<!-- F32: visual order (1→2) differs
     from DOM order (2→1) -->
🔊Screen reader reads"Step 2: Complete the form. Step 1: Read the instructions." — DOM order is wrong; the steps are heard in reverse

Problem (F32): CSS absolute positioning changes the visual position without changing the DOM order. Screen readers read the DOM, so the logical sequence is broken.

✓ Good Practice
HTML + CSSPASS
<!-- C27: DOM order matches
     visual reading order.
     Use CSS flex/grid for layout,
     keep source order logical. -->
<div style="display:flex;gap:1rem">
  <div>
    Step 1: Read the instructions
  </div>
  <div>
    Step 2: Complete the form
  </div>
</div>

<!-- DOM order: 1 → 2 (correct)
     Visual order: 1 → 2 (matches)
     No mismatch. -->
🔊Screen reader reads"Step 1: Read the instructions. Step 2: Complete the form." — correct logical order

Fix (C27): Ensure the DOM source order matches the correct logical reading order. Use CSS Flexbox or Grid to control visual layout while keeping source in the right order. Avoid position: absolute for content ordering.

Example 2 of 4

CSS order Property Creates Reading Order Mismatch

A card component places an image before the heading in the DOM, but uses CSS order in a Flexbox layout to display the heading first visually. This is visually correct but creates a mismatch for screen readers and keyboard users.

Failure: F1 Technique: C27
✗ Poor Practice
HTML + CSSFAIL
<!-- DOM: image first, heading second
     Visual: heading first, image second
     F1: CSS order overrides DOM order -->
<article style="display:flex;
  flex-direction:column">
  <img src="photo.jpg"
       alt="Product photo"
       style="order:2">
  <h3 style="order:1">
    Product Name
  </h3>
  <p style="order:3">Description</p>
</article>

<!-- Screen reader: image → heading → p
     Visual: heading → image → p -->
🔊Screen reader reads"Image: Product photo. Product Name, heading level 3. Description." — image announced before heading creates confusing context

Problem (F1): CSS order visually resequences content but does not change DOM order. Screen reader and keyboard focus order follow the DOM, creating a disconnect from what sighted users experience.

✓ Good Practice
HTMLPASS
<!-- C27: DOM order matches
     the correct reading order.
     No CSS order overrides needed. -->
<article>
  <h3>Product Name</h3>
  <img src="photo.jpg"
       alt="Product photo">
  <p>Description</p>
</article>

/* CSS layout — no order overrides */
article {
  display: flex;
  flex-direction: column;
}
🔊Screen reader reads"Product Name, heading level 3. Image: Product photo. Description." — matches the visual order and logical reading sequence

Fix (C27): Write the DOM in the correct logical order — heading, then image, then description — and use CSS to control layout without changing the sequence. Avoid order, float, or position to swap the reading sequence.

Example 3 of 4

Layout Table Creates Wrong Reading Order

An older page uses an HTML table for two-column layout. The left column contains a navigation sidebar. The right column contains the main article. In the DOM, the navigation cells come before the article cells — but visually users expect article content first.

Failure: F33 Technique: C27
✗ Poor Practice
HTMLFAIL
<!-- F33: layout table creates wrong
     linearised reading order -->
<table>
  <tr>
    <!-- DOM reads nav BEFORE article -->
    <td>
      <nav>
        <a href="#">Section 1</a>
        <a href="#">Section 2</a>
      </nav>
    </td>
    <td>
      <article>
        <h1>Main article heading</h1>
        <p>Article content…</p>
      </article>
    </td>
  </tr>
</table>
🔊Screen reader reads"Navigation, Section 1 link. Section 2 link. Main article heading, heading level 1. Article content…" — nav before main content disrupts reading

Problem (F33): When a layout table is linearised (as screen readers do), the cell contents are read in DOM order — left cell, then right cell — regardless of which column contains the main content.

✓ Good Practice
HTML + CSSPASS
<!-- C27: semantic layout with CSS.
     Main content first in DOM,
     sidebar repositioned with CSS. -->
<div class="page-layout">
  <main>
    <h1>Main article heading</h1>
    <p>Article content…</p>
  </main>
  <nav aria-label="Section navigation">
    <a href="#">Section 1</a>
    <a href="#">Section 2</a>
  </nav>
</div>

/* CSS: nav appears on left visually */
.page-layout { display: flex; }
nav { order: -1; } /* visual only */
🔊Screen reader reads"Main article heading, heading level 1. Article content…" then "Section navigation: Section 1 link. Section 2 link." — main content encountered first

Fix: Replace layout tables with CSS layout (Flexbox, Grid). Put main content before navigation in the DOM. Use order: -1 in CSS to visually move the nav to the left without changing the DOM reading order.

Example 4 of 4

Meaningful Content in CSS ::before / ::after

A developer uses CSS pseudo-elements to insert meaningful text content — for example, adding a "Required" label or numbering steps with counter. This content exists only in CSS and is typically not exposed to screen readers.

Failure: F87 Technique: C22
✗ Poor Practice
CSSFAIL
/* F87: meaningful content in CSS
   ::before — not in DOM, not read
   by most screen readers */

.step::before {
  content: "Step " counter(step) ": ";
  counter-increment: step;
  font-weight: bold;
}

.required-field::after {
  content: " (Required)";
  color: red;
}

/* Screen reader support for
   CSS-generated content is
   inconsistent — never rely on it
   for meaningful information */
🔊Screen reader may announce"Enable notifications" (without step number or required marker) — CSS-generated content is not reliably exposed by assistive technology

Problem (F87): CSS ::before and ::after content is not in the DOM. Screen reader support for generated content is inconsistent — some read it, some do not. Never rely on it for meaningful text.

✓ Good Practice
HTMLPASS
<!-- C22: meaningful text in the DOM.
     CSS for decoration only. -->
<ol>
  <li>
    <label for="notifications">
      Enable notifications
      <span class="req-text">
        (required)
      </span>
    </label>
    <input
      id="notifications"
      type="checkbox"
      required
    >
  </li>
</ol>

/* CSS: style but not generate
   meaningful text */
.req-text { color: red; }
🔊Screen reader reads"List, 1 item. Enable notifications (required), checkbox, required, unchecked." — step sequence from <ol>, required from both text and attribute

Fix: Put meaningful text — step numbers, required indicators, state labels — in the HTML DOM. Use <ol> for numbered sequences. CSS pseudo-elements should only add decorative content (icons, bullets, borders) that carries no meaning.
